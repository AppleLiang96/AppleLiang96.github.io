HashMap源码学习

发表于 2019-07-01 | 分类于 集合源码

HashMap是通过散列表也就是数组+链表（JDK1.8红黑树）的数据结构实现的。他的增删比数组快，查询比链表快。我对HashMap的使用始于刚开始学Servlet做的一个网上书城项目，之前在JavaSE的学习中，一直用的ArrayList，所以一直延续了这个思路，网上书城里有一个业务逻辑应该是从集合里取一个特定的元素，然后发现只能遍历一遍，然后取出该元素，每次取的时间复杂度都是O(n)。当时就很尴尬，然后想到，这个时候我应该用map啊！Anyway，现在把HashMap好好总结一下。

这里又要说一下JDK版本的问题，在1.7版本中，没有用到树的结构，所以当hash冲突很严重的时候，我们使用hashmap查找的时间复杂度会趋近于O(n)，所以1.8版本中，当链表的长度大于8时，就会自动转化为红黑树。

先讲1.7版本的resize方法（1.8版本较难）

这里最难理解的地方就是 e.next = newTable[i]; 这句以及下面这句，这里是怎么做到把链表中的元素转移的呢？链表的头插法，首先e.next= newTable[i]是讲当前节点的next指向了桶中的第1个元素，假设这时候桶中有一个元素a，那么e.next就指向了这个a。newTable[i] = e，这句话就是把e元素放入桶中的第一个元素，原来newTable[i] = a，newTable[i] = e之后指向了e，然后e.next又指向了a，这就是头插链表法。所以这就能理解先放在一个索引上的元素终会被放到Entry链的尾部这句话了。

链表转移中的loNode和hiNode的理解

首先在转移链表的时候，因为数组的长度变了，所以我们的链表中的元素的位置有可能改变（因为hash算法跟长度有关）。看一下经过扩容后再hash有什么改变。

很明显，hash的结果有两种，第一：1&0还是0，所以位置不变。第二1&1是1，那么位置增加一个原来数组的长度（相当于二进制下的10000后面再扩容也有可能是100000）。那么1.8中如何检查这个位置是0还是1呢。继续回上面的resize方法看源码。

HashMap的面试准备

JDK7的put方法
  1. 检查table是否初始化
  2. 如果key是null，把他放到table[0]里面
  3. key不是null，hash（key），并且hash对table.length-1（这里会问为什么要-1）取模找到index也就是table中的索引位置
  4. 遍历链表，查看是否有重复的key，没有就头插（这里会问头插的方法）链表（在插入链表的过程中，先检查hashmap的大小是否达到阈值，如果达到就要扩容，扩容完计算新的下标）
JDK7中的resize(int newCapacity)方法
  1. 首先检查老的长度是不是已经达到了最大上限，如果达到了就把threshold也扩到上限
  2. 没达到上限，就创建一个新的数组，然后把元素都迁移过去。（迁移的过程可能会问高低位迁移的原理）
JDK7中的get方法
  1. 根据hash(key)&(length - 1)计算table的index，找到相应的key（判断相等的方法是==+.equals）
JDK8中的putVal(int hash, K key, V value, boolean onlyIfAbsent, boolean evict)

onlyIfAbsent参数为true说明只有在不存在当前key的时候才进行put操作

  1. 跟JDK7一样。首先判断table是否初始化。
  2. hash&table.length-1找到table下标，如果改下标没有值，那么就初始化node然后放在这个位置。
  3. 如果当前位置有数据，第一步首先判断头结点的key跟要插入的key是否相等，如果相等就替换value，否则的话就开始判断当前节点的类型是红黑树还是链表，这里是插入链表的最后面，如果插入后判断长度大于8还得先变成红黑树。
  4. 如果是覆盖型插入还要返回旧值。
  5. 检查size，如果超过阈值就得扩容
JDK8中的扩容方法resize
  1. 首先判断老的长度是不是已经到达极限了，到达了就把容量threshold扩到极限
  2. 否则长度扩大一倍，容量扩大一倍。
  3. 初始化新的数组，然后开始数据迁移
  4. 主要注意链表的迁移，还是高低位迁移
JDK8中的get方法
  1. 计算 key 的 hash 值，根据 hash 值找到对应数组下标: hash & (length-1)
  2. 判断数组该位置处的元素是否刚好就是我们要找的，如果不是，走第三步
  3. 判断该元素类型是否是 TreeNode，如果是，用红黑树的方法取数据，如果不是，走第四步
  4. 遍历链表，直到找到相等(==或equals)的 key
